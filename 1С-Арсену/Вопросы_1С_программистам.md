**Конфигурация 1С:** Бухгалтерия предприятия 3.0 (3.0.74.76), платформа 8.3.27.1859

## Краткий контекст

Мы внедряем веб-приложение для оперативного учёта по цехам: смены, выработка,
сдельная зарплата, расход материалов. Учёт будет вестись параллельно с вашей 1С,
с двусторонней синхронизацией через **стандартный OData REST API**.

Из 1С нам нужно **читать**: номенклатуру, сотрудников, подразделения, склады.
В 1С мы будем **писать**: документы начисления зарплаты, отчёты производства за
смену, требования-накладные на расход материалов.

Приложение будет развёрнуто **внутри локальной сети Light Company** (не в облаке) —
на собственном сервере компании. Пользователи (бригадиры, директор, бухгалтер)
заходят с планшетов и компьютеров по локальной сети. Выхода данных наружу не будет.

Помогите, пожалуйста, ответить на вопросы ниже — это позволит нам корректно
подключиться без рисков для вашей базы и грамотно подобрать серверную инфраструктуру.

## 4. Доступ к 1С из интернета

Наше веб-приложение задеплоено в облаке (платформа Vercel). 1С у вас скорее
всего стоит внутри офиса и снаружи недоступна — это нормально и безопасно.
Нужно решить, **как именно** мы будем соединяться.

**Какой вариант вам удобнее?** (от простого к сложному)

- [ ] **A. Локальный прокси-сервер** на стороне Light Company.
  Мы напишем маленькую программу (Node.js), которая будет работать у вас на сервере 1С
  (или на любом компьютере той же сети) и проксировать запросы от нашего веб-приложения
  в 1С. Снаружи открывается только один порт через HTTPS, защищённый ключом.

- [ ] **B. Cloudflare Tunnel** (бесплатный сервис).
  Устанавливаем небольшой клиент на ваш сервер, он держит исходящее соединение
  с интернетом. Порты наружу открывать не надо. Технически простой и безопасный вариант.

- [ ] **C. Прямая публикация 1С в интернет** через HTTPS reverse-proxy.
  Если у вас уже есть веб-сервер с белым IP — можно настроить публикацию с
  Basic Auth + ограничение по IP. Требует более серьёзной настройки фаервола.

- [ ] **D. Свой вариант:** \_\_\_\_\_\_\_\_\_\_\_\_\_

**Если не уверены — рекомендуем A или B**, мы поможем настроить.

## 5. Открытие регистров для OData (если понадобится)

По умолчанию OData отдаёт только **справочники** и **документы**. Регистры
накопления (например, фактические остатки) и регистры сведений по умолчанию **закрыты**.

Если нам потребуется их читать (для контроля остатков материалов, для сверок) —
нужно будет однократно запустить **BSL-скрипт в конфигураторе** для открытия
конкретных регистров.

Готовы запустить такой скрипт по нашему запросу? Это короткий код на встроенном

## Что мы делаем после получения ответов

1. Анализируем структуру вашей базы (через `$metadata`)
2. Уточняем точные имена документов и реквизитов
3. Пишем клиент OData с учётом ваших ответов
4. Согласовываем тестовую отправку 1-2 документов
5. После проверки бухгалтером — включаем автоматическую синхронизацию
