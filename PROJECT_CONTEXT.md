## 1. О проекте

**LightFlow** — система оперативного цехового учёта (MES) для производства Light Company (г. Кисловодск, обувь из ЭВА, ПВХ, силикона с 2004 года).

**Главная цель:** заменить бумажные тетради бригадиров на цифровой контур, в котором фиксируются:
- Сменная выработка по артикулам
- Движение партий между цехами
- Расход сырья и брак
- Сдельная зарплата работников

**Чего НЕ делает LightFlow** (это остаётся в 1С Бухгалтерии):
- Бухгалтерский учёт
- Начисление зарплаты и налогов (LightFlow только считает сделку и передаёт сумму)
- Финансовый учёт
- Документооборот с клиентами

## 4. Производственный поток (9 цехов)

```
[1. Сырьё] → [2. Литейка] → [3. УПАКОВКА-ДИСПЕТЧЕР] → ветвление маршрутов:
                                                          ├→ Простые: сразу в [8. Маркировка]
                                                          ├→ Средние: [6. Клеевой] → [7. Обшив] → [8. Маркировка]
                                                          └→ Сложные: [4. Крой] → [5. Швейка] → [6. Клеевой] → [7. Обшив] → [8. Маркировка]
                                                                                                                                ↓
                                                                                                                          [9. Склад]
```

**КЛЮЧЕВАЯ ОСОБЕННОСТЬ:** Упаковочный цех работает в паре с литейкой и является **диспетчером**. Партии после литья попадают в упаковку, где их обрезают, сортируют и распределяют по маршрутам. Это критическая особенность производства, которую нельзя упустить.

## 6. Структура базы данных (12 таблиц)

### Справочники (синхронизируются с 1С через XLSX):

**`articles`** — артикулы продукции
- `id` (uuid PK), `code` (text — например "112/н"), `name` (text), `material` (enum: 'ЭВА', 'ПВХ', 'силикон'), `box_qty` (int — пар в коробке), `size_min` (int), `size_max` (int), `wholesale_price` (numeric)

**`materials`** — сырьё и материалы
- `id` (uuid PK), `code` (text), `name` (text), `unit` (enum: 'кг', 'шт', 'м', 'м²'), `current_stock` (numeric)

**`employees`** — работники
- `id` (uuid PK), `tab_number` (text — табельный из 1С), `full_name` (text), `workshop_id` (fk → workshops), `position` (text), `is_active` (bool), `hire_date` (date)

**`workshops`** — цеха
- `id` (uuid PK), `code` (text), `name` (text), `seq_order` (int — порядок в потоке 1-9), `color` (text — hex цвет)

### Нормативы (ведёт технолог):

**`rates`** — расценки
- `id` (uuid PK), `workshop_id` (fk), `article_id` (fk — nullable, для общих ставок), `operation` (text), `rate_per_unit` (numeric), `unit_type` (enum: 'пара', 'деталь', 'операция', 'единица'), `valid_from` (date), `valid_to` (date — nullable)

**`norms`** — нормы расхода материалов
- `id` (uuid PK), `article_id` (fk), `material_id` (fk), `qty_per_pair` (numeric)

**`routes`** — маршруты движения партий по артикулам
- `id` (uuid PK), `article_id` (fk), `sequence` (jsonb — массив workshop codes), `is_active` (bool)

### Оперативные данные (ядро системы — выделено в схеме):

**★ `shifts`** — смены
- `id` (uuid PK), `workshop_id` (fk), `foreman_id` (fk → employees), `shift_date` (date), `shift_type` (enum: 'день', 'ночь'), `status` (enum: 'open', 'closed'), `opened_at` (timestamp), `closed_at` (timestamp)

**★ `shift_outputs`** — выработка по сменам
- `id` (uuid PK), `shift_id` (fk), `article_id` (fk), `quantity` (int — пар), `weight` (numeric — кг), `defect_qty` (int), `machine` (text), `downtime_min` (int), `batch_id` (fk → batches — после закрытия смены)

**`shift_workers`** — работники в смене
- `id` (uuid PK), `shift_id` (fk), `employee_id` (fk), `operations` (jsonb — что делал), `qty_done` (int), `calculated_pay` (numeric)

**★ `batches`** — партии (сердце системы)
- `id` (uuid PK), `batch_number` (text unique — формат "ЦЕХ-ДДММГГ-NN" например "ЛИТ-110526-01"), `article_id` (fk), `quantity` (int), `weight` (numeric), `created_at` (timestamp), `created_in_workshop` (fk), `current_workshop` (fk), `status` (enum: 'created', 'in_transit', 'received', 'in_work', 'completed', 'shipped'), `defect_total` (int)

**★ `batch_movements`** — движения партий между цехами
- `id` (uuid PK), `batch_id` (fk), `from_workshop` (fk — nullable для первой записи), `to_workshop` (fk), `moved_at` (timestamp), `moved_by` (fk → employees), `qty_in` (int), `qty_out` (int), `defect_at_step` (int), `notes` (text — nullable)

**`material_consumption`** — расход сырья
- `id` (uuid PK), `shift_id` (fk), `material_id` (fk), `qty_used` (numeric), `is_by_norm` (bool — по нормативу или по факту)

### Выгрузка в 1С:

**`payroll_lines`** — строки ведомости
- `id` (uuid PK), `period` (text — "2026-05"), `employee_id` (fk), `workshop_id` (fk), `amount` (numeric), `breakdown` (jsonb — детализация по операциям), `exported_at` (timestamp — nullable)

## 8. Маршруты по типам артикулов

```typescript
// Логика определения маршрута по артикулу:
type Route = 'simple' | 'medium' | 'complex';

function getRoute(article: Article): Route {
  // Сложные: с манжетом, Аляски (АВ-серия) — нужен текстиль
  if (article.code.includes('манжет') || article.code.startsWith('АВ-') || article.code.includes('113')) {
    return 'complex'; // → крой → швейка → клей → обшив → маркировка
  }
  // Простые: галоши, сабо без утеплителя
  if (['сабо', 'галоши'].some(t => article.name.toLowerCase().includes(t))) {
    return 'simple'; // → упаковка → маркировка
  }
  // Средние: сапоги, полусапожки
  return 'medium'; // → клей → обшив → маркировка
}
```

**ВАЖНО:** это базовая логика. Реальные маршруты должен подтвердить технолог через таблицу `routes`. Не хардкодить, использовать как fallback.

## 9. Расчёт сдельной зарплаты

**Формула:** `ЗП работника за смену = Σ (qty_done × rate_per_unit) по каждой операции`

**Логика по цехам:**

| Цех | База расчёта | Пример |
|---|---|---|
| Литьё | руб/пара по артикулу | Сапоги 112/н: 15₽ × 96 пар = 1440₽ |
| Упаковка | руб/пара (общий тариф) | 2₽ × 240 пар = 480₽ |
| Крой | руб/деталь | 1.5₽ × 192 деталей = 288₽ |
| Швейка | руб/операция | Пристрочка манжета: 3₽ × 80 = 240₽ |
| Клеевой | руб/пара | 5₽ × 96 пар = 480₽ |
| Обшив | руб/пара + руб/фурнитура | 4₽ × 96 + 0.5₽ × 384 заклёпки = 576₽ |
| Маркировка | руб/пара | 1₽ × 240 пар = 240₽ |

**Реальные расценки задаёт технолог в таблице `rates`. На старте — заполнить тестовыми значениями.**

## 10. Жизненный цикл партии

```
1. Бригадир литейки в форме смены добавляет выработку:
   "Артикул 112/н, 96 пар, 68 кг, машина ИЛМ-3"
   
2. Бригадир нажимает "Закрыть смену":
   - Из shift_outputs автоматически создаётся batch
   - batch_number формируется: "ЛИТ-ДДММГГ-NN"
   - Партия получает статус 'created'
   - Создаётся batch_movement: from=null, to=Литейка
   
3. Партия отображается в "Входящие" у бригадира Упаковки.
   Он нажимает "Принять":
   - Создаётся batch_movement: from=Литейка, to=Упаковка
   - qty_in (может быть меньше из-за брака), defect_at_step
   - Партия получает status='received'
   
4. Бригадир Упаковки выбирает следующий маршрут:
   "В Клеевой" / "Сразу на Маркировку" / "На доработку"
   - Создаётся batch_movement к следующему цеху
   - Партия становится 'in_transit'
   
5. И так далее по маршруту до Склада.
   На складе: status='shipped' при отгрузке.
```

**Номер партии формируется автоматически:**
- Префикс цеха-создателя (ЛИТ, УПК, КЛЕ, ОБШ, МРК)
- Дата ДДММГГ
- Порядковый номер 01-99 в рамках дня и цеха

## 11. Интеграция с 1С Бухгалтерия

**Принцип:** обмен через XLSX-файлы, никаких прямых API. Бухгалтер вручную выгружает/загружает.

### Импорт из 1С (раз в неделю):
1. **Сотрудники** — Excel с колонками: `tab_number, full_name, position, workshop_code, hire_date, is_active`
2. **Номенклатура** — Excel с колонками: `code, name, material, box_qty, size_min, size_max, wholesale_price`

### Выгрузка в 1С (раз в месяц):
1. **Ведомость зарплаты** — XLSX:
   - `tab_number, full_name, workshop, period, total_amount, breakdown`
2. **Расход материалов** — XLSX:
   - `material_code, material_name, period, qty_used, unit`
3. **Выпуск готовой продукции** — XLSX:
   - `article_code, article_name, period, qty_produced, qty_shipped`

**Формат файлов согласовать с бухгалтером перед реализацией.**

## 13. Правила работы AI-ассистентов

### Для Claude Code (сложные задачи):
- Сначала читай этот файл целиком
- Перед написанием кода — пиши план в чате, жди подтверждения
- Большие задачи разбивай на этапы, коммить каждый этап
- Используй TypeScript строго (никаких `any`)
- Для UI — только shadcn/ui компоненты
- Для БД — только через Supabase client, не прямой SQL в коде
- После каждой задачи — обновляй WORKPLAN.md (отмечай выполненное)

### Для Codex (атомарные задачи):
- Не трогай файлы, которые сейчас редактирует Claude Code (смотри git status)
- Генерируй компоненты по образцу уже существующих
- Не придумывай новые библиотеки — используй только из package.json
- Соблюдай структуру папок из раздела 12

### Общее:
- Все промпты на русском, код на английском (имена переменных, функций)
- Комментарии в коде на русском, если они объясняют бизнес-логику
- Никогда не хардкодить расценки/нормы/маршруты — всё из БД
- Цвета — только из переменных в раздел 3
- Перед любым коммитом — проверка TypeScript и lint

## 14. Чего НЕ делать (anti-patterns)

❌ Не использовать `localStorage` для бизнес-данных — только Supabase
❌ Не делать инлайновые стили — только Tailwind классы
❌ Не запрашивать БД в Client Components — через Server Components или API
❌ Не считать зарплату в UI — только в сервисах (`lib/services/payroll.ts`)
❌ Не использовать `useState` для серверных данных — TanStack Query или Server Components
❌ Не писать длинные функции (>50 строк) — разбивать
❌ Не коммитить `.env` или ключи Supabase
❌ Не использовать AI для генерации деловой логики без проверки человеком (особенно расчёт денег)
