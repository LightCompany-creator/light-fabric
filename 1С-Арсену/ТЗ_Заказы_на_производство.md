## 1. Что такое LightFabric (напоминание)

Программа цехового учёта обувной фабрики Light Company: смены, выработка,
сдельная зарплата, расход материалов, остатки по цеховым складам, перемещения
между цехами. 1С (БП 3.0) остаётся мастером справочников и получателем
итоговых документов — подробности в `00-Прочти-первым.md` и `API_1C_Endpoints.md`.

Этот документ описывает новый слой поверх уже реализованного: **заказ на
производство** — от коммерческого отдела до сдачи готовой продукции и ЗП.

## 2. Бизнес-процесс

```
Коммерческий отдел (стол заказов, дефицит по остаткам — уже есть)
        │  формирует заказ на производство
        ▼
Начальник производства (Арут)
        │  разбивает заказ на подзаказы по цехам, ставит сроки
        ▼
Начальники цехов (Размела / Оля / Нина / Коля / Рома...)
        │  принимают в работу ИЛИ просят скорректировать срок/состав
        │  открывают смены со ссылкой на подзаказ
        │  выработка идёт в счёт подзаказа
        ▼
Подзаказ закрыт (обе стороны подтвердили) → …
… когда закрыты все подзаказы → начальник производства закрывает главный заказ
```

### Иерархия документов

- **Главный заказ** — один номер, приходит от коммерческого отдела через
  начальника производства. Не редактируется цехами напрямую.
- **Подзаказ** — создаётся начальником производства на основе главного заказа,
  один на цех (или несколько, если один цех участвует в заказе дважды).
  Содержит: артикулы + количество, срок выполнения, статус.

### Статусы подзаказа

1. `создан` — начальник производства назначил цеху, срок задан.
2. `на корректировке` — начальник цеха запросил изменение (обычно срока),
   ждёт решения начальника производства.
3. `в работе` — начальник цеха принял, смены цеха ссылаются на этот подзаказ.
4. `закрыт` — по аналогии с уже реализованной моделью перемещений между цехами:
   закрывается, когда **обе стороны подтвердили** выполнение (начальник цеха —
   что сдал, начальник производства — что принял). После закрытия правка
   запрещена, открыть может только админ.

### Статус главного заказа

- `закрыт`, когда закрыты **все** его подзаказы. Закрывает начальник
  производства вручную (не авто) — чтобы он мог свериться перед сдачей
  коммерческому отделу.

### Привязка к сменам и прогресс

- При открытии смены начальник цеха указывает номер подзаказа (если смена
  идёт в счёт заказа; демо-выработку и «фоновую» продукцию вне заказов нужно
  оставить возможной — не весь объём цеха обязательно идёт по заказам).
- По каждому подзаказу видно: **план** (артикул + количество из подзаказа) vs
  **факт** (сумма выработки смен, привязанных к этому подзаказу) → остаток.
- Начальник цеха закрывает подзаказ, когда факт устраивает обе стороны —
  необязательно строго план = факт (перевыполнение/недовыполнение с
  комментарием — обычная практика производства).

## 3. Что нужно построить в LightFabric (сторона приложения, для контекста Арсену)

Это делаем сами, не требует действий от Арсена — but перечисляю, чтобы было
понятно, откуда возникают новые точки обмена с 1С в разделе 4:

- Справочник/реестр «Заказы на производство» (главные) — created either
  вручную начальником производства, либо импортом из внешнего источника.
- Экран разбивки заказа на подзаказы по цехам со сроками.
- В карточке цеха/смены — выбор подзаказа, прогресс план/факт.
- Кнопки «запросить корректировку» / «принять» / «закрыть» с обеих сторон.
- Список «мои подзаказы» у начальника цеха, «мои заказы» у начальника
  производства — аналогично уже существующему списку документов перемещений.

## 4. Интеграция с 1С — что нужно уточнить

Это главный технический вопрос этого документа.

### 4.1 Где живёт «стол заказов»

**Уточнено (22.07.2026): стол заказов дописан прямо в 1С Бухгалтерии** —
это доработка поверх типовой БП 3.0 (типовой функционал такого документа не
содержит, значит объект кастомный). Дефицит по остаткам тоже считается там же.

Раз это доработка, а не типовой объект — Арсену нужно будет описать её
руками, готовых имён объектов из документации БП 3.0 тут не будет. Смотри
вопросы в §6.

### 4.2 Что нужно от Арсена по этой доработке

Добавляем в `API_1C_Endpoints.md` новый пункт обмена (имена объектов и полей —
ориентировочные, Арсен подставит реальные из своей доработки):

**`GET /production-orders?changed_since=`** (1С → LightFabric)
```json
[{
  "code": "00-000789",
  "number": "З-2026-000789",
  "date": "2026-07-20",
  "customer": "...",
  "due_date": "2026-08-01",
  "lines": [
    { "nomenclature_code": "00-00001234", "qty": 500, "unit": "пар" }
  ],
  "status": "Подтверждён"
}]
```

**`POST /production-order-status`** (LightFabric → 1С, опционально) — сообщать
1С, что главный заказ закрыт производством:
```json
{
  "external_id": "uuid",
  "order_code": "00-000789",
  "status": "Выполнен",
  "closed_date": "2026-07-30"
}
```

Подзаказы по цехам (внутренняя разбивка) в 1С **не отражаем** — это чисто
внутренняя механика LightFabric, 1С не обязана про неё знать.

### 4.3 Приоритет

Этот контур делаем **после** уже согласованных этапов 1–2 из
`API_1C_Endpoints.md` (справочники, отчёт производства, требование-накладная).
Заказы — это надстройка над тем, что уже работает, не блокер для запуска
базового обмена.

## 5. Вопрос: может ли LightFabric стать внешней оболочкой для 1С?

Наира попросила рассмотреть — не для решения прямо сейчас, а чтобы держать
в голове при проектировании обмена.

**Что уже происходит:** LightFabric читает справочники из 1С и пишет обратно
итоговые документы (производство, ЗП, накладные). Де-факто для цехов и
планшетов 1С уже не нужна — они целиком работают в LightFabric. С добавлением
заказов эта картина расширяется на начальника производства.

**Куда это реалистично может дорасти:**
- Стол заказов коммерческого отдела сейчас реализован доработкой в самой 1С —
  если коммерческому отделу со временем будет удобнее работать в общем
  интерфейсе с производством, экран стола заказов тоже можно перенести в
  LightFabric, и тогда весь оперативный контур (заказ → производство → склад
  → ЗП) будет жить в одном приложении, а 1С получает из него только готовые
  бухгалтерские документы.
- Директор и бухгалтер тоже могут постепенно меньше открывать саму 1С для
  повседневных задач — дашборды и отчёты уже в LightFabric.

**Где предел, за который лучше не заходить:**
- БП 3.0 — регламентированный бухучёт (налоги, отчётность в ФНС, кадровый
  учёт). Это не место, которое имеет смысл «оборачивать» — легче оставить как
  систему-источник для регламентированной отчётности и передавать туда готовые
  документы, чем переписывать бухгалтерию заново.
  проведение (создание движений по регистрам) при записи через OData всё равно
  выполняется логикой самой 1С — то есть даже при полном переходе интерфейса
  на LightFabric, 1С никуда не девается как движок, который считает бухгалтерию.
- OData 1С не резиновый: много одновременных сессий с планшетов при большом
  штате нагружает базу. Уже сейчас в вопросах к Арсену стоит вопрос про
  очередь/реальное время (см. §5 вопросов в `API_1C_Endpoints.md`) — это то же
  самое соображение.
- Полная замена 1С как движка — это переписать весь бухучёт заново, задача
  совершенно другого масштаба и не то, что нужно фабрике.

**Рекомендация:** LightFabric — это оперативная надстройка над 1С
(«внешняя оболочка» для операционных процессов производства и склада), а не
замена 1С целиком. Расширять эту надстройку постепенно, по мере готовности
(следующий кандидат — стол заказов, если Наира решит его туда перенести), но
1С оставить системой-источником для бухгалтерии, кадров и регламентированной
отчётности. Формально ничего не мешает идти дальше и дальше, но каждый
следующий шаг — отдельное решение, не берём его сейчас как обязательство.
