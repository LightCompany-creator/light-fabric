## 🔴 КРИТИЧНОЕ (P1)

### P1-1 · Уязвимости в зависимостях
- **Где:** package.json
- **Что:** `next` (14.2.35) — 14 высоких CVE: DoS, SSRF, кэш-poisoning, XSS, request smuggling. `xlsx` (0.18.5) — Prototype Pollution и ReDoS, фикса нет.
- **Воспроизведение:** `npm audit --omit=dev`
- **Причина:** Next.js не обновлялся, xlsx архивная.
- **Рекомендация:**
  - Обновить Next до 14.2.40+ (или 15.x с миграцией): `npm i next@latest`
  - xlsx заменить на `@e965/xlsx` (форк без уязвимостей, drop-in) ИЛИ принять риск (используется только на сервере с проверенными входами)

## 🟠 ВАЖНОЕ (P2)

### P2-1 · RLS «расслаблены» для демо
- **Где:** `supabase/migrations/0011_demo_relax_shifts_rls.sql`
- **Что:** Любой аутентифицированный с записью в `employees` может открыть смену в **любом** цехе, добавить выработку, работников, расход материалов. Изначально политика была: бригадир — только свой цех.
- **Когда чинить:** перед боевым запуском вернуть прежнюю логику или сделать гибридную (бригадир — свой; директор/админ — любой).

### P2-2 · Импорт артикулов/сотрудников ломается из роли «бухгалтер»
- **Где:** RLS-политика `tech writes articles` в [supabase/migrations/0002_rls_minimal.sql:46-49](supabase/migrations/0002_rls_minimal.sql#L46-L49)
- **Что:** UI «Импорт из 1С» доступен бухгалтеру, но RLS разрешает запись в `articles`/`employees` только технологу и админу. В журнале `sync_log` это видно: 3 импорта с `failed: 369/97`, ничего не вставилось. Все 369 артикулов и 97 сотрудников залиты через service_role-скрипт.
- **Воспроизведение:** залогиниться `accountant@lightflow.test` → /sync/import → загрузить файл → 0 вставок, 100% ошибок (тихо)
- **Причины:** (1) RLS не покрывает роль accountant; (2) `applyArticlesImport`/`applyEmployeesImport` не показывают ошибки RLS в UI, считают только `failed += 1`.
- **Рекомендация:** Либо расширить RLS на accountant ("tech+acc writes articles"), либо в UI показывать конкретные ошибки RLS вместо общего числа.

### P2-3 · 342 артикула без норм расхода (88%)
- **Где:** таблица `norms`
- **Что:** Из 388 артикулов нормы заданы только для 46. При закрытии смены расход материалов спишется только для тех артикулов, у которых есть нормы — остальные не попадут в `material_consumption`.
- **Рекомендация:** Технолог должен пройтись по новым 342 артикулам и задать нормы расхода. Или хотя бы базовая логика по материалу (ЭВА → 0.7 кг на пару по умолчанию).

### P2-4 · `console.error` в продакшене
- **Где:** [app/(app)/shifts/actions.ts:51](app/(app)/shifts/actions.ts#L51)
- **Что:** `console.error("[openShiftAction] failed:", err);` — единственный console-вывод. Безобидно для логов, но для прода лучше пробросить в Sentry или подобный.

## 🟡 СРЕДНЕЕ (P3)

### P3-1 · Schema drift: код базы вручную поправлен `0009`–`0011`, а `schema.sql` обновлён частично
- **Где:** [schema.sql](schema.sql), [supabase/migrations/0009_drop_batch_tracking.sql](supabase/migrations/0009_drop_batch_tracking.sql), [0010](supabase/migrations/0010_add_cast_forms.sql), [0011](supabase/migrations/0011_demo_relax_shifts_rls.sql)
- **Что:** `schema.sql` приведён в соответствие после удаления партий и добавления `cast_forms`, но не отражает изменения RLS из 0011. Свежий запуск schema.sql даст не ту RLS, что сейчас в БД.
- **Рекомендация:** Перегенерировать `schema.sql` из текущего состояния БД (`pg_dump --schema-only`) перед боевым запуском.

### P3-2 · Расценки заведены, но не для всех 388 артикулов
- **Где:** таблица `rates`
- **Что:** Из 388 артикулов per-article расценки только для 163. Остальные 225 будут считать ЗП по общецеховой ставке (LIT 10 ₽/пара) — потенциально неточно.
- **Рекомендация:** Технолог уточняет расценки для остальных категорий (фото 2 — рекламная маркировка, фото 4 — корректировки 22.02.26 не применены).

### P3-3 · 17 кодов из фото с расценками не нашлись в каталоге
- **Где:** `scripts/apply-rates.mjs`
- **Что:** `044`, `4023`, `315`, `6023`, `811`, `1033`, `223`, `665`, `161`, `063`, `080/1`, `086/1`, `067`, `058`, `062` — нет таких артикулов в БД. `058` и `062` это операции «манжет» а не артикулы (стоит завести как общецеховые расценки с operation).

### P3-4 · `_migrations` таблица без RLS
- **Где:** создаётся в [scripts/run-migrations.mjs:18-25](scripts/run-migrations.mjs#L18-L25)
- **Что:** Служебная таблица учёта применённых миграций. RLS на ней не настроен — service_role обходит и так, но если кто-то получит anon-доступ, увидит список миграций (минимальная утечка инфы).
- **Рекомендация:** `alter table _migrations enable row level security;` без политик — всем закрыт.

### P3-5 · Лишние таблицы `product_lines` миграций 0005-0006
- **Где:** [supabase/migrations/0005_product_lines.sql](supabase/migrations/0005_product_lines.sql), [0006](supabase/migrations/0006_articles_product_line.sql)
- **Что:** Введены раньше «направления продукции», но в коде (`/catalog/*`) нигде не используются. Возможно мёртвый функционал.
