## 2. Уточнённая модель производства (запись Наиры 11.06.2026 — источник истины)

> ⚠️ **Бригадиров НЕТ. Везде начальники цехов.** Терминология в UI — «начальник цеха».

### Поток производства

```
Литьё:  ЭВА-гранулы (чёрная / цветная; вес обуви зависит от цвета)
        → Литейка (Размела) → литьё
        → в основном Аруту (Маркировка/Упаковка), реже сразу на склад ГП

Крой:   Рубка/Крой = ОДНО И ТО ЖЕ (Оля) → заготовки (носки, верха «Аляски»)
        → Швейка (Нина) — сборка/обшив заготовок
        → Обшив (Оля, совмещает) — манжеты, вставки, носки
        → в основном Аруту, в крайних случаях сразу склад ГП

Клеевой: полуфабрикаты с Литейки → Клеевой (Коля)
        → лейблы, проклейка, кроксы → Аруту

Финал:  Маркировка/Упаковка (Арут — нач. всего производства)
        → маркировка (штрихкод) ИЛИ упаковка размерным рядом
        → «Склад Кисловодск» (Рома)
```

### Структура цехов в БД vs реальная

| В системе (код) | Что внутри | Начальник цеха |
|---|---|---|
| ANG («Ангар») | **Откладываем!** Цех химии + склад сырья | Мурат, Халил |
| LIT (Литейка) | Литьё из ЭВА (чёрная/цветная гранула) | Размела |
| PACK (Упаковка) | На практике — часть Маркировки (Арут). **Уточнить как использовать** | — |
| CUT (Рубка/Крой) | Рубка и крой — одно и то же. Заготовки: носки, верха «Аляски» | **Оля** |
| SEW (Швейка) | Сборка/обшив заготовок от кроя | Нина |
| GLU (Клеевой) | Лейблы, проклейка, кроксы. **Джибитсы УБИРАЕМ** (решено) | Коля |
| ASSY (Обшив) | Манжеты, вставки, носки (как у галош) | **Оля** (совмещает с кроем) |
| MARK (Маркировка) | Маркировка (штрихкод) или упаковка размерным рядом. Маркетплейс-операции | **Арут** (он же начальник всего производства) |
| SHIP (Склад) | **«Склад Кисловодск»** — склад готовой продукции | Рома |
| ADM (Администрация) | Бухгалтерия, директор. Оставить как «цех» в системе | — |
| LST (Листы) | ЭВА-листы, коврики, автонакидки, стропа, тесьма. **Вернёмся позже** | — |

### Склады и перемещения (главное требование — реализовать первым)

1. **У каждого цеха свой склад.** Сколько цехов — столько подразделений и складов.
2. Произвели → продукция попадает **на остаток склада своего цеха**.
3. Передача = **документ перемещения**: отправитель создаёт → получатель
   проверяет и принимает → остаток списывается у отправителя и приходуется
   у получателя.
4. После принятия обеими сторонами документ **закрывается** — редактирование
   запрещено. Открыть закрытый документ может **только админ**.
5. Непринятые документы можно свободно корректировать и удалять.
6. ЗП: начальник цеха при внесении выработки указывает, **кто произвёл**
   (уже реализовано через смены).

### Общие принципы учёта (по umbrella-doc)
1. **Смена** суточная, день/ночь, открывает бригадир/начальник цеха
2. **Выработка** = работник × модель × операция × количество → расценка → ЗП
3. **Передача между цехами = перемещение с подтверждением приёма** (см. §3 решение)
4. **Внешние приходы** заводятся в 1С → потом в цеха
5. **Внутренние перемещения** ведут цеха сами
6. **Расход материалов** автоматически: по нормам или по весам (литьё)
7. **Контроль остатков** — нельзя сдать больше чем выработано
8. **Синхронизация с 1С** — параллельная, в реальном времени (см. §5)

## 3. Решения по требованиям производства (зафиксированные ответы Наиры)

### Документоориентированный подход (§1 ответа Наиры)
- Любой документ можно **корректировать и удалять**
- Документ перемещения между цехами после подтверждения обеими сторонами →
  **закрывается**. Открыть может только **админ** (override).
- Начальник цеха видит «свой список документов за выбранную дату».
- На основе документа смены создаются производные документы:
  производство → ЗП → передача.

### Структура цехов и склады (§2 ответа Наиры)
- Каждый цех имеет **свой склад** с остатками
- На складе: полуфабрикаты + готовая продукция + комплектующие (молнии, клей, бирки)
- Фильтрация по типу (полуфабрикаты / готовая / комплектующие)
- Перемещения идут не строго по цепочке, а между любыми цехами

### Открытые вопросы (отложены до момента «увидим как работает»)
- **Цвет гранулы** — отдельные артикулы или поле в выработке (вес обуви зависит от цвета)
- **Учёт станков** — справочно или нет
- **Глубина учёта материалов** — какие позиции учитывать
- **PACK (Упаковка)** — как использовать: на практике часть Маркировки

## 5. План интеграции с 1С

### Известные параметры 1С Light Company
- **Платформа:** 1С:Предприятие 8.3.27.1859
- **Конфигурация:** Бухгалтерия предприятия 3.0 (3.0.74.76)
- **Юрисдикция:** РФ (ИП Хачатуров)
- **Способ интеграции:** OData REST API (стандартный)

### Архитектура (план по образцу проекта Алексея)
Алексей (выпускник 15-го потока, владелец фасадной компании) сделал MCP-сервер
для 1С УНФ. Его опыт и промт сохранены в `Aleksey/`. Используем его подход
с адаптацией под БП 3.0.

**Два возможных подхода:**

**A. Прямой OData-клиент** в Next.js (`lib/services/onec.ts`)
- Server Action после записи в Supabase шлёт HTTP в 1С
- Простой, без отдельного процесса
- ✅ Рекомендуется

**B. MCP-сервер** для 1С (как у Алексея)
- Отдельный процесс
- Может использоваться и нашим приложением, и Claude-агентом
- Сложнее
- Может пригодиться позже для AI-аналитики

### Что синхронизировать (БП 3.0)

**Из 1С → LightFabric** (читаем):
- `Catalog_Номенклатура` → артикулы
- `Catalog_Сотрудники` / `Catalog_ФизическиеЛица` → работники
- `Catalog_Подразделения` → цеха (соответствие кодов)
- `Catalog_Склады` → склады
- `Catalog_Организации` → юр. лица

**Из LightFabric → 1С** (пишем):
- `Document_НачислениеЗарплаты` ← `payroll_lines` (по закрытию месяца)
- `Document_ОтчётПроизводстваЗаСмену` ← `shifts` + `shift_outputs` (по закрытию смены)
- `Document_ТребованиеНакладная` ← `material_consumption`

### Этапы реализации
1. **Шаг 1** — получить от программистов 1С ответы на вопросы из документа
2. **Шаг 2** — заглушка `lib/services/onec.ts` с интерфейсом, возвращающая stub
3. **Шаг 3** — реальный OData-клиент когда узнаем параметры (URL, тех. пользователь)
4. **Шаг 4** — мост между Vercel/локальным сервером и 1С (см. §6)
5. **Шаг 5** — двусторонняя синхронизация в реальном времени
